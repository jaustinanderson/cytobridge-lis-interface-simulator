# CytoBridge v1.1 requirements - Controlled Error-Queue Recovery

> **DRAFT FOR AUSTIN REVIEW.**
> These requirements are Claude's draft derived solely from the frozen
> [v1.1 design record](v1.1-design-record.md). They describe observable
> behavior only and add no architecture beyond that record. They are **not
> frozen** and describe behavior that is **not yet implemented**. They become
> frozen only after Austin approves and merges the setup pull request. Do not
> read these as evidence that v1.1 behavior exists.

Numbered, testable requirements for CytoBridge v1.1 recovery. IDs continue the
existing sequence after `R-019` (Sessions 1-3, see
[`requirements.md`](requirements.md)). Each statement describes a behavior the
simulator is expected to demonstrate once implemented, within the project's
synthetic scope - **no PHI**, not a validated medical device or production LIS.

## How to read a requirement

| Field | Meaning |
|---|---|
| **ID** | Stable identifier (`R-020` ...) |
| **Statement** | The observable behavior expected of the simulator |
| **Source** | Design-record section that authorizes it |
| **Type** | `Data model`, `Recovery workflow`, `State model`, `Idempotency`, `Service`, or `Audit` |

## Structured failure classification

| ID | Statement | Source | Type |
|---|---|---|---|
| **R-020** | Each error-queue item carries a structured `failure_code`, a `failure_category` that is one of the five approved categories (`MESSAGE_STRUCTURE`, `ORDER_MATCHING`, `ORDER_STATE`, `SPECIMEN`, `FISH_RESULT_CONTENT`), a `recovery_policy` of `RETRY_OR_REDRIVE`, `REDRIVE_ONLY`, or `TERMINAL`, and retains the existing detailed analyst-readable `reason`. | Sec 5.1, 6 | Data model |
| **R-021** | Every existing failure maps to exactly the approved failure-code/category/policy triple in the design record's mapping table, and no code, category, or policy outside that table is produced; an unmappable failure is surfaced as a blocker rather than assigned a new value. | Sec 6 | Data model |

## Immutability and lineage

| ID | Statement | Source | Type |
|---|---|---|---|
| **R-022** | The original failed `interface_message` (its `message_id`, payload, control ID, status, and timestamps) and the queue item's `raw_payload` copy are never modified by any recovery action. | Sec 4, 5.1, 5.3 | Data model |
| **R-023** | The original failed message is never updated to `FILED`; only a newly created recovery message may reach `FILED`. | Sec 4 | Data model |
| **R-024** | Each recovery attempt is recorded in `interface_recovery_attempt` with the approved logical fields, links to its `interface_error_queue` item via `queue_id`, and, when processing occurs, to the resulting `interface_message` via `resulting_message_id`; retried and corrected messages are stored only in the existing `interface_message` table. | Sec 4, 5.2, 5.3 | Data model |

## Unchanged retry

| ID | Statement | Source | Type |
|---|---|---|---|
| **R-025** | An unchanged retry (`RETRY_ORIGINAL`) is permitted only for an `OPEN` queue item whose failure is `ORDER_NOT_FOUND`; it is rejected for any other failure or policy. | Sec 7 | Recovery workflow |
| **R-026** | A permitted unchanged retry processes an exact copy of the immutable original payload, creates a distinct new `interface_message` row, and records that new message on the recovery attempt. | Sec 7 | Recovery workflow |

## Corrected re-drive

| ID | Statement | Source | Type |
|---|---|---|---|
| **R-027** | A corrected re-drive (`REDRIVE_CORRECTED`) is permitted for an `OPEN` queue item whose policy is `REDRIVE_ONLY` or `RETRY_OR_REDRIVE`, and is rejected for a `TERMINAL` item. | Sec 7 | Recovery workflow |
| **R-028** | A permitted corrected re-drive processes the supplied corrected payload, creates a distinct new `interface_message` row without overwriting the original payload or the original queue copy, and records that new message on the recovery attempt. | Sec 7 | Recovery workflow |

## Terminal failures

| ID | Statement | Source | Type |
|---|---|---|---|
| **R-029** | Any recovery request against a `TERMINAL` queue item (including `ORDER_FINALIZED` and `ORDER_CANCELLED`) is rejected; recovery never reopens, overrides, unfinalizes, or uncancels an order. | Sec 7, 8 | Recovery workflow |

## Queue state model

| ID | Statement | Source | Type |
|---|---|---|---|
| **R-030** | An error-queue item's `status` is one of `OPEN`, `RESOLVED`, or `TERMINAL`, and the only automatic transitions are `OPEN -> RESOLVED` (after one successful recovery) and `OPEN -> TERMINAL`; a `RESOLVED` or `TERMINAL` item never automatically returns to `OPEN`. | Sec 5.1, 8 | State model |
| **R-031** | Queue timestamp state is consistent: `OPEN` has null `resolved_at` and null `terminal_at`; `RESOLVED` has a populated `resolved_at` and null `terminal_at`; `TERMINAL` has a populated `terminal_at` and null `resolved_at`. | Sec 5.1 | State model |

## Recovery-attempt outcomes and history

| ID | Statement | Source | Type |
|---|---|---|---|
| **R-032** | Each recovery attempt records an `outcome` of `SUCCEEDED` (processing filed, queue resolved), `FAILED` (processing failed; the resulting message is preserved with status `ERRORED` and the queue item remains `OPEN`), or `REJECTED` (action prohibited or queue already closed); `SUCCEEDED` and `FAILED` attempts have a resulting message and `REJECTED` attempts create no resulting processing message. | Sec 5.2, 8 | State model |
| **R-033** | Recovery-attempt history for a queue item is retrievable and reflects every attempt and its outcome in order. | Sec 2, 10 | Audit |

## Idempotency and atomicity

| ID | Statement | Source | Type |
|---|---|---|---|
| **R-034** | `request_id` is globally unique; re-submitting an existing `request_id` whose queue item, action, payload fingerprint (`payload_sha256`), and actor all match the originally recorded request returns the existing recorded outcome and creates or changes no records. | Sec 5.2, 9 | Idempotency |
| **R-035** | A queue item has at most one `SUCCEEDED` recovery attempt, enforced independently of `request_id` uniqueness. | Sec 5.2, 9 | Idempotency |
| **R-036** | After a queue item is `RESOLVED`, a further recovery request with a new `request_id` is `REJECTED`: no processing message is created, no FISH result changes, and no additional filing event occurs. | Sec 8, 9 | Idempotency |
| **R-037** | A genuinely `FAILED` attempt may be retried later with a new `request_id` while the queue item remains `OPEN` and the action is allowed. | Sec 9 | Recovery workflow |
| **R-038** | Each permitted recovery operation commits or rolls back as a unit: on success the resulting message, FISH result changes, audit events, recovery-attempt outcome, and queue transition all commit together; on a handled recovery failure the FISH-result changes, filing events, and queue resolution are rolled back, while the resulting message is preserved with status `ERRORED`, the recovery attempt is preserved with outcome `FAILED` and its explanation (`outcome_detail`), and the queue item remains `OPEN`. No other partial state persists. | Sec 9 | Recovery workflow |
| **R-041** | Re-submitting an existing `request_id` whose queue item, action, payload fingerprint (`payload_sha256`), or actor does not match the originally recorded request is rejected as a `REQUEST_ID_CONFLICT`: no message, recovery attempt, or FISH result is created, the original attempt record is never overwritten, and the conflict is recorded in the existing audit trail. | Sec 5.2, 9 | Idempotency |

## Service boundary and visibility

| ID | Statement | Source | Type |
|---|---|---|---|
| **R-039** | Recovery is exposed as a headless Python service layer providing retry, corrected re-drive, and recovery-history behavior equivalent to `retry_queue_item(...)`, `redrive_queue_item(...)`, and `get_recovery_history(...)`, with no web UI, REST API, full CLI, or unrelated framework. | Sec 10 | Service |
| **R-040** | Each recovery attempt and its outcome produce audit evidence and troubleshooting-visible detail (including `payload_sha256` and `outcome_detail`) sufficient to explain what payload was attempted and why it succeeded, failed, or was rejected. | Sec 5.2, 9 | Audit |
