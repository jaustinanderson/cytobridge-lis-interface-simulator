# CytoBridge v1.1 test-intent specification

> **DRAFT FOR AUSTIN REVIEW.**
> This is Claude's pre-implementation test intent derived solely from the frozen
> [v1.1 design record](v1.1-design-record.md) and the draft
> [v1.1 requirements](v1.1-requirements.md). It describes *what a test would
> assert*, not executable tests, and contains **no fixtures generated from
> application output**. It is **not frozen** and does not imply v1.1 behavior
> exists. It becomes frozen only after Austin approves and merges the setup pull
> request. Per the design record, expected-output fixtures for the human-approved
> invariants must be reviewed by Austin against the design record and may not be
> back-filled from an implementation run.

Each entry states: starting state, action, expected changed data, expected
unchanged data, and where applicable the expected message / queue /
recovery-attempt / FISH-result / audit state, repeated-action behavior, and
explicit negative and rollback expectations. All data is synthetic - **no PHI**.

## Requirement-to-intent map

### R-020 - Structured failure classification present

- **Starting state:** An inbound message fails and is routed to
  `interface_error_queue`.
- **Action:** Inspect the queue item.
- **Expected changed data:** The item exposes `failure_code`,
  `failure_category` (one of the five approved categories), `recovery_policy`
  (one of `RETRY_OR_REDRIVE`, `REDRIVE_ONLY`, `TERMINAL`), and retains a
  detailed `reason`.
- **Expected unchanged data:** The original message and its `raw_payload` copy.
- **Negative:** No queue item exposes a category, code, or policy outside the
  approved sets.

### R-021 - Exact approved failure mapping

- **Starting state:** One failed queue item exists per existing failure kind.
- **Action:** Read each item's `failure_code`, `failure_category`,
  `recovery_policy`.
- **Expected:** Each triple equals exactly the design-record mapping row for
  that failure (e.g. `ORDER_NOT_FOUND` -> `ORDER_MATCHING` /
  `RETRY_OR_REDRIVE`; `ORDER_FINALIZED` -> `ORDER_STATE` / `TERMINAL`;
  `EMPTY_MESSAGE` -> `MESSAGE_STRUCTURE` / `REDRIVE_ONLY`).
- **Negative:** A failure that cannot map is surfaced as a blocker, never
  assigned an invented code, category, or policy.

### R-022 / R-023 - Original-message and queue-payload immutability

- **Starting state:** A failed original message and its queue item.
- **Action:** Perform any permitted recovery (retry or re-drive).
- **Expected unchanged data:** Original `message_id`, payload, control ID,
  status, timestamps, and queue `raw_payload` are byte-for-byte identical
  before and after.
- **Expected changed data:** A new `interface_message` row exists.
- **Negative:** The original message never reaches `FILED`; only the new
  recovery message may become `FILED`.

### R-024 - Recovery-attempt lineage

- **Starting state:** An `OPEN` recoverable queue item.
- **Action:** Perform a permitted recovery.
- **Expected changed data:** A new `interface_recovery_attempt` row links to the
  queue item (`queue_id`) and, because processing occurred, to the resulting
  `interface_message` (`resulting_message_id`).
- **Expected unchanged data:** No recovery-payload store exists outside the
  existing `interface_message` and queue records.

### R-025 / R-026 - Unchanged retry eligibility and behavior

- **Starting state:** An `OPEN` queue item with failure `ORDER_NOT_FOUND` and a
  now-available matching order.
- **Action:** `RETRY_ORIGINAL`.
- **Expected changed data:** A new `interface_message` containing an exact copy
  of the immutable original payload; the attempt records that new message;
  processing files results to the matched order; queue transitions
  `OPEN -> RESOLVED`.
- **Expected unchanged data:** Original message and queue payload.
- **Negative:** `RETRY_ORIGINAL` against any non-`ORDER_NOT_FOUND` item is
  `REJECTED` with no new processing message.

### R-027 / R-028 - Corrected re-drive eligibility and distinct storage

- **Starting state:** An `OPEN` queue item with policy `REDRIVE_ONLY` or
  `RETRY_OR_REDRIVE`.
- **Action:** `REDRIVE_CORRECTED` with a corrected payload.
- **Expected changed data:** A new distinct `interface_message` row holding the
  corrected payload; the attempt records that new message.
- **Expected unchanged data:** Original payload and original queue copy are not
  overwritten.
- **Negative:** `REDRIVE_CORRECTED` against a `TERMINAL` item is `REJECTED`.

### R-029 - Terminal failure rejection

- **Starting state:** A `TERMINAL` queue item (`ORDER_FINALIZED` or
  `ORDER_CANCELLED`).
- **Action:** Any recovery request.
- **Expected:** `REJECTED`; no resulting processing message; the order is not
  reopened, unfinalized, or uncancelled.
- **Rollback:** No partial state; the queue item remains `TERMINAL`.

### R-030 / R-031 - Queue states, transitions, timestamp consistency

- **Starting state:** `OPEN` items.
- **Action:** Drive one to a successful recovery and one to a terminal
  condition.
- **Expected changed data:** `OPEN -> RESOLVED` sets `resolved_at`, leaves
  `terminal_at` null; `OPEN -> TERMINAL` sets `terminal_at`, leaves
  `resolved_at` null.
- **Negative:** No automatic `RESOLVED -> OPEN` or `TERMINAL -> OPEN`
  transition; `OPEN` has both timestamps null.

### R-032 / R-033 - Attempt outcomes and history

- **Starting state:** A recoverable queue item.
- **Action:** Produce a `SUCCEEDED`, a `FAILED`, and a `REJECTED` attempt across
  appropriate scenarios.
- **Expected:** `SUCCEEDED` and `FAILED` attempts carry a resulting message;
  `REJECTED` carries none; `FAILED` leaves the queue `OPEN`; `SUCCEEDED`
  resolves it. History retrieval returns every attempt and outcome in order.

### R-034 - Request idempotency

- **Starting state:** A successful recovery under `request_id A`.
- **Action:** Re-submit `request_id A`.
- **Expected:** The existing recorded outcome is returned; no message, FISH
  result, attempt, or audit record is created or changed.

### R-035 / R-036 - One success per item; rejection after resolution

- **Starting state:** A `RESOLVED` queue item.
- **Action:** Submit a new `request_id B`.
- **Expected:** `REJECTED`; no processing message, no FISH result change, no
  second filing audit event; exactly one `SUCCEEDED` attempt exists for the
  item.

### R-037 - Failed attempt may be retried with a new request_id

- **Starting state:** An `OPEN` item with a prior `FAILED` attempt.
- **Action:** Submit a new `request_id` for an allowed action.
- **Expected:** A fresh attempt is processed (no idempotency short-circuit)
  because the item is still `OPEN` and the `request_id` is new.

### R-038 - Atomic commit or rollback

- **Starting state:** A permitted recovery that is forced to fail mid-operation.
- **Action:** Attempt the recovery.
- **Expected rollback:** None of the resulting message, FISH result changes,
  audit events, attempt outcome, and queue transition persist; the database is
  left in its pre-attempt state. On success all of them persist together.

### R-039 - Headless service boundary

- **Starting state:** N/A (interface shape).
- **Action:** Exercise recovery through the Python service functions
  (`retry_queue_item`, `redrive_queue_item`, `get_recovery_history` or
  repository-conventional equivalents).
- **Expected:** Behavior is reachable directly with no web UI, REST API, full
  CLI, or added framework.

### R-040 - Audit and troubleshooting visibility

- **Starting state:** Any recovery attempt.
- **Action:** Inspect audit evidence and troubleshooting output.
- **Expected:** `payload_sha256` and `outcome_detail` explain which payload was
  attempted and why the outcome occurred; successful filing produces the same
  class of audit event the inbound path already emits.

## Human-approved invariants (copied without weakening)

The following two invariants are reproduced from the frozen design record
(section 11). They may be added to but not weakened or replaced. Expected-output
fixtures representing them must be reviewed by Austin against the design record
and must not be back-filled from an implementation run.

### Invariant I-01 - Original-message immutability

The test must:

1. Create a failed message and snapshot the original message ID, payload,
   control ID, status, timestamp, and queue payload.
2. Perform a corrected re-drive.
3. Prove every snapshotted original value remains unchanged.
4. Prove the corrected payload has a distinct message ID.
5. Prove the recovery attempt links the queue case to the new message.
6. Prove only the new message can become `FILED`.

### Invariant I-02 - Duplicate and replay protection

The test must:

1. Create one recoverable queue item.
2. Recover it successfully using `request_id A`.
3. Snapshot messages, FISH results, recovery attempts, and filing audit events.
4. Submit `request_id A` again.
5. Prove the existing outcome is returned with no new or changed records.
6. Submit a different `request_id B` against the resolved queue item.
7. Prove the request is `REJECTED`, no processing message is created, no FISH
   result changes, and no second filing event occurs.
8. Prove exactly one successful recovery exists for the queue item.
