## 2. Product definition

**CytoBridge v1.1 - Controlled Error-Queue Recovery** adds a safe, headless recovery workflow for failed synthetic AML/MDS FISH inbound messages.

The workflow must let an analyst:

1. Inspect a failed message and its error-queue record.
2. Request an allowed retry of the unchanged payload or a corrected re-drive.
3. Preserve the original failed message permanently.
4. Prevent duplicate or repeated successful recovery.
5. Record every recovery attempt and outcome.
6. Resolve the queue item after exactly one successful recovery or mark it terminal when recovery is prohibited.

This remains an educational, synthetic, intentionally headless portfolio project. It is not a production clinical application, certified interface engine, validated medical device, or Epic/Beaker implementation.

## 4. Approved recovery history model

The original failed message is immutable. It remains stored in `interface_message` with its original `message_id`, payload, control ID, status, and timestamps.

Every requested recovery creates or reuses an `interface_recovery_attempt` record. When processing is permitted, the attempt points to a new `interface_message` row:

- An unchanged retry creates a new message row containing an exact copy of the original payload.
- A corrected re-drive creates a new message row containing the corrected payload.
- The recovery-attempt record links the error-queue item, original message indirectly through that queue item, and resulting message.
- A rejected request creates an attempt record but no resulting processing message.

The original message is never updated to `FILED`. Only a new recovery message may reach `FILED`.

## 5. Approved target data model

### 5.1 `interface_error_queue` target additions

The existing queue remains the single recovery case. It gains:

| Field | Purpose |
|---|---|
| `failure_code` | Stable machine-readable failure code from the approved mapping. |
| `failure_category` | One of the five approved categories. |
| `recovery_policy` | `RETRY_OR_REDRIVE`, `REDRIVE_ONLY`, or `TERMINAL`. |
| `status` | Expanded to `OPEN`, `RESOLVED`, or `TERMINAL`. |
| `terminal_at` | Timestamp when an open item becomes terminal. |

Existing `reason` remains the detailed analyst-readable explanation. Existing `raw_payload` remains an immutable copy of the original payload. Existing `resolved_at` timestamps successful resolution.

State and timestamp consistency must be enforced:

- `OPEN`: `resolved_at` and `terminal_at` are null.
- `RESOLVED`: `resolved_at` is populated and `terminal_at` is null.
- `TERMINAL`: `terminal_at` is populated and `resolved_at` is null.

### 5.2 New `interface_recovery_attempt` table

The target table contains exactly these logical fields:

| Field | Purpose |
|---|---|
| `attempt_id` | Integer primary key. |
| `queue_id` | Required foreign key to `interface_error_queue`. |
| `resulting_message_id` | Nullable foreign key to the new `interface_message` created when processing occurs. |
| `request_id` | Required caller-supplied idempotency identifier; globally unique. |
| `action` | `RETRY_ORIGINAL` or `REDRIVE_CORRECTED`. |
| `payload_sha256` | SHA-256 fingerprint of the payload associated with the request. |
| `outcome` | `SUCCEEDED`, `FAILED`, or `REJECTED`. |
| `actor` | Required free-text actor consistent with the existing educational identity model. |
| `outcome_detail` | Human-readable explanation of the outcome. |
| `attempted_at` | Attempt timestamp. |

Database enforcement must include:

- Unique `request_id`.
- At most one row with `outcome = 'SUCCEEDED'` for a given `queue_id`.
- Valid enumerated action and outcome values.
- `SUCCEEDED` and `FAILED` attempts require a resulting message.
- `REJECTED` attempts do not create a resulting processing message.
- Valid foreign keys.

No third recovery table is approved.

### 5.3 Existing `interface_message`

Retried and corrected messages continue to use the existing `interface_message` table. The recovery attempt provides lineage. No duplicate recovery payload store is approved outside the existing message and queue records.

## 7. Approved recoverability rules

### Retry unchanged

An unchanged retry is approved only for `ORDER_NOT_FOUND`, after the required matching order may have become available.

The retry:

- Uses an exact copy of the immutable original payload.
- Creates a distinct new `interface_message` row.
- Records the new message on the recovery attempt.

### Corrected re-drive

A corrected re-drive is allowed for `REDRIVE_ONLY` and `RETRY_OR_REDRIVE` queue items.

The re-drive:

- Accepts a corrected payload.
- Creates a distinct new `interface_message` row.
- Never overwrites the original payload or original queue copy.
- Records the new message on the recovery attempt.

### Terminal failures

`ORDER_FINALIZED` and `ORDER_CANCELLED` are terminal. Recovery must be rejected. CytoBridge v1.1 may not reopen, override, unfinalize, or uncancel an order.

## 8. Approved state model

### Queue states

- `OPEN`: eligible for an approved recovery action.
- `RESOLVED`: one recovery succeeded.
- `TERMINAL`: recovery is permanently prohibited.

Allowed automatic transitions are only:

- `OPEN -> RESOLVED`
- `OPEN -> TERMINAL`

`RESOLVED` and `TERMINAL` never return to `OPEN` automatically.

### Attempt outcomes

- `SUCCEEDED`: processing filed and the queue item resolved.
- `FAILED`: processing failed; the resulting message is preserved as `ERRORED`
  and the queue item remains open.
- `REJECTED`: the action was prohibited or the queue was already closed.

No asynchronous `QUEUED`, `SCHEDULED`, or `PROCESSING` states are approved because the simulator is synchronous and single-process.

## 9. Approved idempotency and transaction rules

Two separate protections are required:

1. **Request idempotency:** `request_id` is globally unique. Repeating an existing `request_id` returns the existing recorded outcome and performs no new processing only when the queue item, action, payload fingerprint (`payload_sha256`), and actor all match the original request. Any mismatch is rejected as a `REQUEST_ID_CONFLICT`: it creates no message, recovery attempt, or FISH result, never overwrites the original record, and creates only an event in the existing audit trail.
2. **Business invariant:** a queue item may have no more than one successful recovery attempt.

`payload_sha256` provides audit evidence about which payload was attempted but is not the sole idempotency control.

Each permitted recovery operation commits or rolls back as a unit. On success, the resulting message, FISH result changes, audit events, recovery-attempt outcome, and queue transition all commit together. A handled recovery failure rolls back the FISH-result changes, filing events, and queue resolution, while preserving the resulting message as `ERRORED`, the recovery attempt as `FAILED` with its explanation, and the original queue item as `OPEN`.

A genuinely failed attempt may be tried again later using a new `request_id`, provided the queue item remains `OPEN` and the requested action is allowed.

## 10. Approved service boundary

Recovery remains headless and is implemented as a small Python service layer, expected to provide functions equivalent to:

- `retry_queue_item(...)`
- `redrive_queue_item(...)`
- `get_recovery_history(...)`

Exact Python names and signatures may follow existing repository conventions, but the builder may not replace this boundary with a web UI, REST API, full CLI, or unrelated framework.

Tests, the demo, and troubleshooting documentation call or describe the service directly.

## 11. Human-approved invariant tests

These expected behaviors are approved before implementation. Claude may add coverage but may not weaken or replace these assertions.

### Invariant I-01 - Original-message immutability

The test must:

1. Create a failed message and snapshot the original message ID, payload, control ID, status, timestamp, and queue payload.
2. Perform a corrected re-drive.
3. Prove every snapshotted original value remains unchanged.
4. Prove the corrected payload has a distinct message ID.
5. Prove the recovery attempt links the queue case to the new message.
6. Prove only the new message can become `FILED`.

### Invariant I-02 - Duplicate and replay protection

The test must:

1. Create one recoverable queue item.
2. Recover it successfully using `request_id A`.
3. Snapshot messages, FISH results, recovery attempts, and filing audit events.
4. Submit `request_id A` again with the identical request parameters (queue
   item, action, payload fingerprint, and actor).
5. Prove the existing outcome is returned with no new or changed records.
6. Submit a different `request_id B` against the resolved queue item.
7. Prove the request is `REJECTED`, no processing message is created, no FISH result changes, and no second filing event occurs.
8. Prove exactly one successful recovery exists for the queue item.

Expected-output fixtures may not be back-filled from an implementation run and automatically accepted. Austin must review any fixture representing these invariants against this record.
